# 1.0 Release Audit

Audit date: 2026-06-04

This audit maps the 1.0 feature set to current evidence. It is intentionally about release readiness, not the longer 1.1/1.2 visual roadmap.

## Verdict

Status: implementation-complete for the 1.0 substrate, pending final release verification.

The automated contract is strong: the local acceptance gate covers lint, full Python tests with 100% coverage, dynamic-port dogfood smoke, browser screenshot rendering, replay baselines, hard-coded renderer baselines, whitespace checks, and tracked secret/runtime hygiene. The remaining 1.0 work is release process and one real live dogfood pass with harn/Codex, not a known missing framework boundary.

## External Tester Handoff

This branch is a 1.0 test candidate, not a gold release. A reviewer should try to break the 1.0 substrate and report any missing framework boundary, not polish the long-term visual direction.

Start from a clean checkout:

```bash
git pull --ff-only
uv sync
uv run harn-gibson --version
bash scripts/acceptance.sh
```

For a quick non-live smoke that avoids provider/auth state:

```bash
uv run harn-gibson dogfood \
  --harn-bin true \
  --no-browser \
  --no-codex-auth-import \
  --no-hold-on-error
```

For visual review without a live harn session:

```bash
uv run harn-gibson watch-replay examples/gibson1-replays/repo-map-trajectory.json \
  --renderer-command 'uv run python examples/renderers/gibson1_renderer.py' \
  --renderer-timeout-ms 10000 \
  --step-delay-ms 250 \
  --max-step-delay-ms 1000
```

For the required live dogfood pass before gold, use a scratch workspace outside tracked source files:

```bash
mkdir -p test-artifacts/dogfood-workspaces/manual-1.0
uv run harn-gibson dogfood \
  --cwd test-artifacts/dogfood-workspaces/manual-1.0 \
  -- -p "Create a tiny Python project with one test, run it, make one small change, rerun the test, and summarize the result."
```

What to inspect:

- The browser opens as the primary Gibson display, with debug details hidden until toggled.
- Browser input queues and reaches harn as follow-up or steering input.
- Runtime/auth/harn failures appear in the debug drawer rather than disappearing into terminal-only output.
- Stream updates do not flood the renderer-agent boundary.
- Renderer context contains project metadata, repo topology, touched files, visual continuity, render intents, and `context.project.worldModel`.
- The world model includes durable file, command, change, health, and outcome facts with provenance.
- Replay fixtures remain deterministic with committed baselines and nonblank screenshots.
- Raw captures, auth files, `.env` files, API keys, and local runtime config stay out of git.

Treat these as 1.0 blockers if found:

- harn cannot launch with the Gibson extension configured from `dogfood`.
- The display server cannot receive harn events or publish browser SSE updates.
- Browser input cannot be delivered through the extension queue.
- Renderer failures are silent or block harn progress instead of failing open with diagnostics.
- Replay baselines or screenshot expectations fail without an intentional baseline-changing source edit.
- The world-model payload is absent from renderer context or loses provenance for derived facts.
- Any runtime auth/config/key material appears in tracked files.

## Remaining 1.0 Release Work

1. Run one short real dogfood session with harn/Codex, not just the `--harn-bin true` smoke. The purpose is to catch provider/auth/harn packaging issues that unit tests and smoke mode intentionally avoid.
2. Rerun `bash scripts/acceptance.sh` after any release-candidate edits.
3. Decide release metadata:
   - package version for 1.0;
   - changelog or release notes;
   - tag name and whether to push the tag.
4. Do a final docs pass for first-time users:
   - normal dogfood path;
   - renderer implementation path;
   - primitive/backend implementation path;
   - replay review workflow.
5. Confirm `git status --short --branch` is clean and only intended tracked files are in the release commit.

## Deferred After 1.0

These are valuable, but they should not block the 1.0 substrate release:

- Provider-SDK-backed renderer agent.
- Semantic code graph and test-to-code mapping.
- Richer agent attention/intent signals from harn.
- Stale/reconciled world-model facts.
- More coherent default visual direction beyond current `gibson1`.
- Lower-level binding vocabulary for world-model-backed spatial primitives.
- Alternate non-web display backend implementation.
