# 1.0 Feature Set

`harn-gibson` 1.0 is the stable substrate for a Hollywood-style harn display. The release should be dogfoodable against a live harn session, replayable offline, and explicit about the renderer-agent contract. It does not need the final creative model loop to be impressive yet; it does need every boundary that loop will use to be real, tested, and documented.

Current release-readiness evidence is tracked in [1.0-release-audit.md](1.0-release-audit.md).

## Release Goal

1.0 ships a primary browser display for harn with:

- live harn event capture through a project extension;
- a browser input queue that can send lightweight follow-up or steering input back to harn;
- a persistent scene engine with mutation primitives, animations, event logs, tracebacks, and hook decisions;
- an event-sourced renderer world model with bounded file/activity/change/command/health/outcome facts;
- a renderer routing layer that can choose renderer-agent, local stream buffer, debug-only, direct-scene, sampled, or drop paths;
- blocking and async render pipeline modes;
- replay fixtures that can reproduce agent-side events, renderer plans, and explicit scene mutations;
- full local test coverage, browser screenshot integration tests, and secret hygiene checks.

## Public Surface

The 1.0 contract is the shape downstream renderers, test fixtures, and dogfood sessions can rely on:

- Commands: `serve`, `run`, `capture`, `import-codex-auth`, `backend-contract`, `catalog`, `replay`, `watch-replay`, `replay-dir`, and `event-log-to-replay`. `dogfood` and `dogfood-capture` remain compatibility aliases for `run` and `capture`.
- Harn commands: `/gibson-view` starts or reuses a browser viewer from an interactive harn session; `/gibson-renderers` lists built-in viewer renderer presets.
- Version metadata: `harn-gibson --version` reports the package version and is tested against `pyproject.toml`.
- Environment knobs: `HARN_GIBSON_ENDPOINT`, `HARN_GIBSON_EVENT_LOG`, `HARN_GIBSON_HOOKS`, `HARN_GIBSON_PROJECT_ROOT`, `HARN_GIBSON_PROJECT_NAME`, `HARN_GIBSON_RENDER_MODE`, `HARN_GIBSON_RENDER_BATCH_MS`, `HARN_GIBSON_RENDER_TIMING`, `HARN_GIBSON_RENDERER_INTEREST`, `HARN_GIBSON_RENDERER_COMMAND`, `HARN_GIBSON_RENDERER_TIMEOUT_MS`, `HARN_GIBSON_RENDERER_MODEL_COMMAND`, `HARN_GIBSON_RENDERER_MODEL_TIMEOUT_MS`, `HARN_GIBSON_RENDERER_COMPACTION_EVENTS`, `HARN_GIBSON_RENDERER_MAX_RECENT_PLANS`, `HARN_GIBSON_RENDERER_MAX_RECENT_LOG_ENTRIES`, `HARN_GIBSON_RENDERER_MAX_PROP_PREVIEW_CHARS`, `HARN_GIBSON_RENDERER_MAX_VISUAL_ANCHORS`, `HARN_GIBSON_RENDERER_MAX_VISUAL_OBJECTS_PER_ANCHOR`, `HARN_GIBSON_RENDERER_MAX_VISUAL_RECENT_ITEMS`, `HARN_GIBSON_RENDERER_MAX_REPO_ENTRIES`, `HARN_GIBSON_RENDERER_MAX_REPO_CHILDREN`, `HARN_GIBSON_RENDERER_MAX_TOUCHED_FILES`, `HARN_GIBSON_RENDERER_MAX_TOUCHED_PATH_CHARS`, `HARN_GIBSON_RENDERER_MAX_WORLD_ENTITIES`, `HARN_GIBSON_RENDERER_MAX_SEMANTIC_FILES`, `HARN_GIBSON_RENDERER_MAX_SEMANTIC_EDGES`, `HARN_GIBSON_RENDERER_MAX_SEMANTIC_SYMBOLS`, `HARN_GIBSON_ROUTE_RULES`, and `HARN_GIBSON_STYLE`.
- Browser/display endpoints and offline catalog discovery: `/`, `/health`, `/scene`, `/catalog`, `/backend-contract`, `/events`, `/events/stream`, `/input`, `/input/next`, and `uv run harn-gibson catalog`.
- Event schema: `GibsonEvent` dictionaries with stable sequence, timestamp, source, type, phase, title, summary, payload, recent-context, and visualization-context fields.
- Scene schema: `SceneState` dictionaries with persistent primitives, animations, bounded event log, revision, and metadata including bounded render-intent history.
- Mutation schema: `upsert`, `patch`, `remove`, `append_log`, `start_animation`, `stop_animation`, and `reset_scene`.
- Renderer schema: `RenderRequest`, `RenderInputBatch`, `RendererContext`, `RenderPlan`, delayed `RenderStep` entries, render intents, renderer event-interest declarations, and explicit route decisions.
- World model schema: `harn-gibson.world-model.v1` with observed per-file activity, structured change facts, command previews, command status/duration, inferred test/build health checkpoints tied to observed command outcomes, source, operation, outcome, revision, truncation, and provenance facts in renderer context.
- Renderer prompt schema: `harn-gibson.renderer-prompt.v1` provider-neutral system/user messages generated from captured renderer contexts for prompt-command or future provider-backed renderer adapters.
- Renderer diagnostics schema: `harn-gibson.render-plan-diagnostics.v1` warnings/errors for external/model renderer plans that are unsupported, unsafe, or would fail scene application.
- Display backend contract: `harn-gibson.display-backend-contract.v1` exposes endpoint paths, scene/update schemas, primitive/effect support, supported mutation ops, input delivery modes, render timing modes, the `harn-gibson.style-pack.v1` schema, supported style packs, the active style pack, and the `harn-gibson.backend-capability-profile.v1` payload for alternate display backends.
- Replay schema: event, raw-event, render-plan, explicit-mutation steps, split event-log manifests, suite review manifests, and final-scene expectations that can produce scene JSON, replay-result JSON, renderer-context JSON, chunked renderer-context/prompt JSON, renderer-prompt JSON/review pages, render-intent JSON/review pages, timeline keyframes, and browser screenshots.

Anything outside that surface can change before 1.0 if it improves the implementation without breaking the contracts above.

## Must Ship

### Dogfood Integration

- `uv run harn-gibson run` starts the display server, opens the browser, imports Codex auth into harn's user auth store when available, and launches harn with the Gibson extension configured.
- `uv run harn-gibson run --cwd PATH` launches harn in an existing target project directory while injecting this repo's extension and Codex model defaults explicitly; renderer context and repo-city visuals use that target directory as the project root.
- `uv run harn-gibson capture` starts the same live session with the hard-coded showcase renderer, writes normalized JSONL to an ignored artifact path, and prints the follow-up replay-review command, including a split-review form when `--split-every` is set.
- `uv run harn-gibson capture --list-trajectories` lists built-in long capture recipes; `--trajectory tiny-project` and `--trajectory repo-map` create ignored bare workspaces, inject long prompts, capture every normalized harn event, and default to split replay review so 15-20 minute dogfood trajectories can become redacted replay and screenshot fixtures without manual setup.
- The display server and extension remain independent of `harn-tui`; only the launcher shells out to harn.
- Runtime diagnostics from the launcher, auth import, harn delivery, and extension failures are visible in the browser debug drawer.
- Browser input is queued on the display server and delivered through the harn extension without pretending the browser owns the harn session.

### Event And Hook Surface

- Normalize harn events into `GibsonEvent` objects with stable `before`, `during`, `after`, and `lifecycle` phases.
- Preserve recent agent context and visualization context fields for future renderer prompts.
- Support hook decisions at the harn mutation points already exposed by harn: input, tool call, tool result, message end, agent preflight, and session-before events.
- Render hook decisions as display/debug inputs without making the renderer responsible for policy enforcement.

### Scene Engine

- Maintain a persistent `SceneState` with primitives, animations, and a bounded log.
- Support the v1 mutation set: `upsert`, `patch`, `remove`, `append_log`, `start_animation`, `stop_animation`, and `reset_scene`.
- Keep tracebacks and runtime failures inspectable as scene state, not only terminal output.
- Provide a deterministic renderer fallback that converts events into useful baseline mutations.

### Browser Display

- Render the Gibson display as the primary experience, not a secondary debug page.
- Keep the debug drawer collapsed by default and make it reopenable/closable.
- Expose health, current scene, catalog, input, and SSE event endpoints.
- Handle desktop and mobile viewport smoke tests without text overlap or blank rendering.

### Renderer Routing And Timing

- Route noisy stream events into local text buffers instead of sending every token-sized update to a remote renderer.
- Let renderers advertise event interest and let route rules sample noisy event types so uninterested or low-value events can fall back to local direct-scene, debug-only, or drop routes.
- Preserve timing in render input batches so a renderer can create effects that cover the same time span as the coalesced events; expose immediate and scheduled playback modes for render-step timing.
- Expose a framework-owned world-model context with bounded entity count and provenance so renderers can bind visuals to durable facts instead of rebuilding memory from recent batches.
- Persist render-intent history so renderer/debug/replay tooling can inspect recent visual decisions.
- Validate external/model render plans before scene application; preserve warning-only plans with diagnostics and reject unsafe plans into deterministic fallback plus trace/debug state.
- Support blocking mode for deterministic ordering and async mode for non-blocking harn progress.
- Expose a visual primitive/effect catalog usable in renderer prompts.

### Replay And Testing

- Replay normalized events through routing and rendering.
- Convert captured normalized JSONL event logs into replay fixtures with default sensitive-value redaction metadata.
- Split long captured JSONL event logs into replay fixture directories with a capture manifest.
- Launch capture-oriented dogfood sessions without manually exporting renderer and event-log environment variables.
- Replay saved renderer plans directly against scene state.
- Replay explicit scene mutations for effect fixtures.
- Watch replay fixtures through the live browser display with configurable step pacing, selected step ranges, expectation controls, and explicit renderer selection.
- Write final scene snapshots and full replay results as JSON.
- Let replay fixtures assert expected final scene state for regression checks.
- Run replay fixture directories as a suite for acceptance checks and per-fixture screenshots.
- Compare replay fixture directories against committed final-scene baselines.
- Export renderer contexts, chunked renderer-context/prompt artifacts, renderer-prompt review artifacts, render-intent review artifacts, split event-log manifests, trajectory coverage summaries, complete single-replay review bundles, and suite-level review bundles from replay runs.
- Render replayed final scenes through the browser display for screenshot review, including checked-in agent-side and renderer-side fixtures.
- Include checked-in dogfood-renderer trajectory fixtures that run the hard-coded showcase renderer against fixture workspaces with committed baselines, fixture project metadata, and browser screenshot expectations.
- Let replay fixtures assert screenshot canvas metrics when browser screenshots are captured, so checked-in visual fixtures can fail on blank or severely underlit rendering.
- Keep Python package coverage at 100%.
- Keep Playwright browser tests available for screenshot review.
- Ensure generated artifacts, runtime auth, API keys, and local config do not enter commits; this is enforced by a tracked-file hygiene test, redacted event-log conversion for promoted fixtures, and the final manual scan.

### Documentation

- Document normal dogfood usage.
- Document manual server/extension usage.
- Document renderer-agent input/output contracts, context strategy, timing, and compaction.
- Document replay fixture format and 1.0 acceptance criteria.

## Explicitly Deferred

- A production upstream harn fork or patch release.
- A polished multi-theme renderer marketplace.
- Multi-client collaborative display control.
- Cloud persistence or remote display hosting.
- Replacing harn's terminal UI as the authoritative session owner.
- A fully creative model-backed renderer as the only renderer path. The 1.0 contracts must support it, but deterministic rendering remains the reliable default.

## 1.1 Direction

1.1 should make the display meaningfully more Gibson-like without changing the 1.0 harness contract. The likely scope is a model-backed renderer adapter behind the existing `SceneRenderer` interface, richer visual primitives/effects, stronger render-intent review, and replay baselines that compare visual state across fixture runs. The first richer primitive pass keeps `mesh`, `spatial_map`, `hologram`, `signal_scope`, `tunnel_grid`, `wire_landscape`, `terminal_wall`, `access_matrix`, `orbital_map`, `data_vault`, `black_ice`, `trace_route`, `svg_layer`, `city_block`, `node_graph`, `ribbon`, `glyph_layer`, `data_rain`, and `particle_field` renderable in the browser so renderer-agent output has visible low-level toys immediately. `spatial_map` adds a lower-level world-binding map for typed objects, edges, stable `entityId` values, focus state, confidence rings, and object-addressable camera targets. `hologram` adds animated projection rings, scan planes, beams, panels, and motes as one structured primitive. `signal_scope` adds a radar/oscilloscope instrument for telemetry, intrusion pings, command activity, and stream summaries, with browser-local sweeps, blip pulses, spokes, rings, and waveforms. `tunnel_grid` adds an animated perspective data corridor for mainframe flythroughs, with browser-local rings, spokes, lanes, packet motes, twist, depth, and inward/outward motion. `wire_landscape` adds a 90s wireframe terrain/filesystem plane with browser-local peak influence, rails, packet motes, labels, and focus state for repo-topology flyovers. `terminal_wall` adds animated banks of bounded terminal panes for command, output, traceback, stream, and touched-file summaries without routing every text delta through a renderer model. `access_matrix` adds animated lock/security grids for modules, files, hook decisions, and breached access cells without per-frame grid mutations. `orbital_map` adds a spinning uplink globe with projected nodes, arcs, rings, packets, scan wedges, and focus pulses for repo or network routes. `data_vault` adds a rotating wireframe mainframe vault with nested cube layers, orbiting locks, scanner panels, and packet motes for access targets or ICE lockboxes. `black_ice` adds a faceted security barrier with scanner shutters, sentry locks, breach glow, and fracture rays for failed commands, runtime errors, or dramatic access gates. `trace_route` adds animated hop-by-hop route lines with packet pulses, curved links, labels, and focus state for network traversal or filesystem navigation scenes. `city_block` now supports browser-sampled `cameraPath` transform keyframes so Gibson filesystem districts can drift like a moving 3D scene. Structured `svg_layer` rendering now supports paths, path morph frames, rects, lines, polylines, polygons, grouped transforms, numeric transform keyframes, named gradients, bounded filter/clip presets, vector-space trace particles, and curated SVG-style symbols such as globes, filesystem gates, reticles, data tunnels, ICE walls, and mainframe cores without accepting raw SVG markup. `data_rain` adds browser-animated glyph curtains for code rain, telemetry waterfalls, and packet noise without model-authored per-glyph mutations. Persistent browser-rendered scene animations cover the first generic effect set: phase pulses, packet bursts, timeline cues, route traces, scans, glitches, signal interference overlays, breach waves, camera jolts, scene camera paths, flythrough rays, extrusion frames, and hold brackets; temporary animations may now include `ttlMs` or `expiresAtMs` so the scene engine can prune stale effects before future renderer contexts. Camera effects can also use `targetRef` to frame named objects inside city blocks, spatial maps, trace routes, node graphs, ribbons, and wire landscapes instead of targeting only whole primitives, and renderer contexts now expose bounded visual-continuity object anchors with exact `targetRef` candidates, focus/tone/metric summaries, and matching world-binding metadata. The hard-coded `gibson1_renderer.py` is now the default `dogfood` renderer preset with a smaller coherent scene, semantic repo city blocks, semantic repo terrain, import/test trace-route links, low-opacity data rain, and style-aware tones, while `gibson_dogfood_renderer.py` remains the broad primitive/effect stress renderer before the model-backed renderer is ready. The first render-intent review pass is `--output-render-intents` plus `--render-intent-review`, which makes renderer decisions and exact timelines inspectable outside the debug drawer. The first renderer-prompt review pass is `--output-render-prompts` plus `--render-prompt-review`, which turns captured renderer contexts into exact provider-neutral messages. The first renderer-agent adapter is the external renderer command contract; the first prompt-backed adapter is `PromptedModelRenderer` plus `HARN_GIBSON_RENDERER_MODEL_COMMAND`, which exercises prompt generation, model-output parsing, plan validation, and fail-open diagnostics before a provider SDK is wired. External/model render-plan validation now rejects unsafe plans before scene application, checks `svg_layer` upserts and patches for raw markup, unbounded keyframes, or unbounded path morphs, warns on unsupported vector filter/clip presets, and records warning-only unsupported choices as render metadata, giving a future creative renderer room to experiment without silently crashing dogfood sessions. The first perception upgrades after the world model are `harn-gibson.agent-attention.v1`, an inferred action/objective/focus payload that helps renderers show what the agent appears to be trying to do, and `harn-gibson.semantic-repo-graph.v1`, a bounded Python package/file/symbol/import/test graph for dependency-aware city layout and causality routes.

`gibson1_renderer.py` now dogfoods object-addressable camera effects directly by aiming subtle camera drift and focus jolts at touched repo-city objects with `targetRef`. It also emits a compact world-model `spatial_map` that binds file object mass, tone, and opacity to accumulated activity, outcome, and lifecycle facts while routing semantic test/import edges between stable `entityId` objects.

## 1.2 Direction

1.2 should push the toy from "instrumented display" toward "cinematic scene engine": stronger animation timelines, higher-level style packs, fuller historical-session replay, bounded repo-topology context with touched-file batches, and a renderer-compaction loop that can preserve visual continuity across long agent sessions. The first timeline step is scheduled render-plan playback, where `startOffsetMs` can drive delayed visual playback across a coalesced event window. `timeline_cue` animations give renderer plans one persistent record for labeled beat markers across that window, while `route_trace` animations give them one persistent waypoint path for packets and traversal across the same window. Structured vector keyframes, path morph frames, city camera keyframes, camera jolts, scene camera paths, wire-landscape packet motion, terminal-wall scan/cursor loops, access-matrix sweeps, orbital-map packet/orbit motion, signal-interference jitter overlays, and filter/clip presets provide lower-level ways to fill those windows with looped drift, scale, rotation, opacity, terrain traversal, text-bank activity, access-board scanning, uplink routing, signal breakup, shape pulsing, iris reveals, wipes, chromatic splits, ghost echoes, and scanlines without asking the model to emit raw SVG markup. The first style-pack step is `HARN_GIBSON_STYLE` plus `--style`, with `gibson`, `neon-noir`, `mainframe`, and `satellite-uplink` packs flowing into scene metadata, browser palette/backdrop behavior, health output, replay screenshots, and renderer-agent context. Browser-local backdrop motifs now make style packs visibly distinct: city packet routes, neon horizon streaks, mainframe phosphor/audit frames, and satellite orbital/radar/chevron effects can render from the style metadata alone. The first repo-scene step maps bounded depth-2 repo topology into Gibson-style city and wire-terrain views, using numeric line-count metadata plus visible file/directory counts for building/peak height and touched files for focus, recolor, particle, extrusion, terrain, terminal-panel, access-cell, uplink-node, and camera drift effects; file contents are never included in that context. World-model lifecycle metadata now gives long sessions a shared current/recent/aging/stale and open/reconciled vocabulary, so renderers can fade older facts without rebuilding their own decay model. The `repo-map` capture trajectory is the first repeatable live-session recipe aimed specifically at collecting depth-2 topology, line-count, command/output, and touched-file evidence for that city/terrain/panel/grid/uplink view. Renderer contexts now include a compact `visualContinuity` payload with current anchors, active animations, recent effects/targets, and style motifs so rolling contexts can preserve visual identity without resending full scene JSON. Renderer context budget environment knobs now tune compaction cadence, recent-plan history, scene preview text, visual anchors, repo entries, repo children, semantic graph file/edge/symbol counts, touched-file count, and touched-path length, so long dogfood sessions can trade prompt size against scene continuity without code edits. The first historical-session step is `--output-timeline`, which captures per-step scene keyframes plus source timing for replay review and later renderer chunking without changing final-scene baselines; timeline and frame screenshot artifacts now include derived source-time summaries for timed/untimed steps, duration, and accumulated delays; `--timeline-screenshot-dir` renders those keyframes as browser screenshots with a source-timed interactive frame-player review page for visual inspection, while `watch-replay` plays full fixtures or selected step ranges through the live browser display with explicit renderer selection, optional source timestamp pacing, and partial-playback expectation controls. Captured event logs can now be split with `event-log-to-replay --split-every --output-dir`, producing chunked replay directories plus `harn-gibson.event-log-split.v1` manifests so 15-20 minute dogfood sessions can be reviewed without one huge fixture. Adding `--review-dir` to that split conversion directly writes `harn-gibson.replay-suite-review.v1` suites with aggregate metrics, trajectory coverage signals/gaps, and links into every chunk's frame player, renderer prompts, renderer chunks, render intents, final scene, and result JSON; `replay-dir --review-dir` remains available for later review passes over the same generated directory. `--output-render-contexts` now captures exact renderer-context payloads from replay, making compaction mode, catalog summaries, visual-continuity anchors, render-input timing, repo topology, semantic graph samples, and touched-file batches reviewable without a live model renderer; `--output-render-chunks` groups those contexts and exact prompts into feedable historical-session batches; `--output-render-prompts` and `--render-prompt-review` capture the corresponding model-message fixture; `--output-render-intents` and `--render-intent-review` capture the corresponding renderer decision history; single-replay `--review-dir` bundles all of those review surfaces into one historical-session directory with a top-level overview.
